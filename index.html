APIConnect

APIConnect provides a very easy and intuitive way to interact with APIs, especially those hosted on other domains. Setup is as simple as creating the instance, specifying the domain, and connecting the route:

var api = new APIConnect();
api.domain('api.twitter.com');
api.connect('statuses/home_timeline');

After this is done the route can be called through the instance:

api.getHomeTimeline();
> GET http://api.twitter.com/statuses/home_timeline.json

The route is automatically set up and accessible through a camelized method name in the api object. Calling the route is as simple as calling the method.

When making the call, the first object is params. This will most commonly be added to the end of the query string, but in cases where the route contains params, it will be placed here instead and removed from the query string:

api.connect('statuses/home_timeline');
api.getHomeTimeline({ foo: 'bar' });
> GET http://api.twitter.com/statuses/home_timeline?foo=bar

api.connect('tweets/:tweet_id');
api.getTweet({ tweet_id: 7 });
> GET http://api.twitter.com/tweets/7

The second object passed is an options hash. Any valid option for jQuery.ajax is also valid here. This means that all the standard callbacks will work:

api.getHomeTimeline({}, {
  success: function() {
    // Congratulations!
  },
  error: function() {
    // Oh noeee... failure!
  }
});

API calls also pass back jQuery deferred objects, so these can be used as well:

api.getHomeTimeline().then(function() {
  // You're done!
});

Any route can be set up with any level of context. These contexts are always optional, and will only be added if they exist when passed as params:

api.connect(':user/:list_id/members');
api.getMembers({ user: 'bob', list_id: 5 });

> GET http://api.twitter.com/bob/5/members.json

The 4 main HTTP verbs, "GET", "POST", "PUT", and "DELETE", are allowed in routes, and map to "get", "create", "update", and "destroy", respectively. "GET" is the implied default:

api.connect('POST :user/:list_id/members');
api.connect('DELETE :user/:list_id/members');

api.createMember({ user: 'bob', list_id: 5 });
> POST http://api.twitter.com/bob/3/members

api.destroyMember({ user: 'bob', list_id: 5 });
> DELETE http://api.twitter.com/bob/5/members

"resource" serves as a shortcut to all 4 HTTP verbs:

api.resource('member');

api.getMember();     > GET     http://api.twitter.com/member.json
api.createMember();  > POST    http://api.twitter.com/member.json
api.updateMember();  > PUT     http://api.twitter.com/member.json
api.destroyMember(); > DELETE  http://api.twitter.com/member.json

If a resource is a collection, passing collection: true in the options for "resource" will create standard collection routes including an "index" method:

api.resource('status', { collection: true });

api.getStatus();              > GET     http://api.twitter.com/status.json
api.getStatus({ id: 3});      > GET     http://api.twitter.com/status/3.json
api.createStatus();           > POST    http://api.twitter.com/status.json
api.updateStatus({ id: 3 });  > PUT     http://api.twitter.com/status/3.json
api.destroyStatus({ id: 3 }); > DELETE  http://api.twitter.com/status/3.json

If a "collection" is omitted, it will attempt to be intelligently detected by the pluralization of the resource passed. In this case, pluralization of the methods will also use intelligent detection:

api.resource('members');

api.getMembers();              > GET     http://api.twitter.com/members.json
api.getMembers({ id: 3});      > GET     http://api.twitter.com/members/3.json
api.createMembers();           > POST    http://api.twitter.com/members.json
api.updateMembers({ id: 3 });  > PUT     http://api.twitter.com/members/3.json
api.destroyMembers({ id: 3 }); > DELETE  http://api.twitter.com/members/3.json
